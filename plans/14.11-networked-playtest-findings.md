## Live blockers

### F1 — `npm run server:dev` fails to resolve `boardgame.io/server` under `tsx`

**Where:** [../server/index.ts:14](../server/index.ts#L14) — `import { Server } from 'boardgame.io/server';`.

**Symptom:** `tsx server/index.ts` throws

```
Error [ERR_MODULE_NOT_FOUND]: Cannot find module
'E:\github2\bgio\node_modules\boardgame.io\server\index.jsx'
imported from E:\github2\bgio\server\index.ts
```

**Diagnosis:** Probed during this session:

- `boardgame.io@0.50.2` does **not** declare an `exports` field
  (probed via `node -e "console.log(require('boardgame.io/package.json').exports)"`
  → `undefined`). It uses the legacy `main` / `module` pair plus
  per-subpath `package.json` files such as
  `node_modules/boardgame.io/server/package.json`:

  ```json
  { "name": "boardgame.io/server", "main": "../dist/cjs/server.js" }
  ```

- Plain Node ESM should follow that `main` to
  `node_modules/boardgame.io/dist/cjs/server.js`. tsx 4.20's
  resolver instead falls into its own `resolveExtensions` path
  (visible in the stack trace at
  `node_modules/tsx/dist/esm/index.mjs:resolveExtensions`) and
  appends `.jsx` first — landing on the non-existent
  `node_modules/boardgame.io/server/index.jsx`.

- Vitest's boot smoke test in
  [`tests/server/boot.test.ts`](../tests/server/boot.test.ts)
  passes because Vite uses its own resolver that respects bgio's
  legacy `main` field correctly.

- `node --import tsx server/index.ts` reproduces the exact same
  failure — tsx is the resolver in both cases.

**Why this is a real blocker (not just a dev-loop nit):** the same
script is invoked by `npm run dev:full`, by `dev:seed` (via
`scripts/dev-seed.ts` which imports server modules indirectly),
and by the CI workflow `deploy-server.yml`. None of those have
been exercised since `npm install` last ran cleanly. The
production Docker build (`server/Dockerfile`) compiles to JS, so
it dodges the issue, but the dev loop is dead until this is
fixed.

**Tried in this session — none worked:**

- `node --import tsx server/index.ts` — same failure (tsx is
  the resolver in both invocations).
- Re-export shim from `server/bgioServer.ts` importing
  `boardgame.io/dist/cjs/server.js` directly. Cleared the first
  failure but the next chained import (`boardgame.io/core` from
  `src/game/resources/moves.ts`) hit the same bug. There are
  **27 files in `src/`** that import from a bgio subpath
  (`/core`, `/ai`, `/client`, `/multiplayer`, `/server`),
  spanning every role/move file. Patching all 27 is not a
  workaround.

**Proposed fix shapes (none yet shipped):**

1. **Pin `tsx` to a known-good version.** The most direct fix.
   Try `tsx@4.7.x`, last verified known-good with bgio 0.50.x
   layout in similar projects. Requires `npm install` on the
   test host.
2. **Replace `tsx` with `vite-node` in the `server:dev`
   script.** Vite's resolver already handles bgio correctly
   (the boot smoke test in
   [`tests/server/boot.test.ts`](../tests/server/boot.test.ts)
   passes via Vite). Adds `vite-node` as a devDependency.
3. **Build the server first** (`npm run server:build` already
   exists) and run the compiled JS with plain Node. Slower dev
   loop but no extra deps.
4. **File the bug upstream against tsx.** When tsx encounters a
   bare specifier whose target is a directory containing a
   `package.json`, it should read that file's `main` field
   before extension-shopping; instead it appends `.jsx` first
   and fails. Symptom is reproducible with bgio 0.50.2 + tsx
   4.20.0 on Node 20.19.5 / Windows.

Option 2 is most robust (matches what tests already do).
Option 1 is fastest if a known-good pin exists.

A separate sub-plan ([14.14](14.14-tsx-bgio-resolver.md)) tracks
the fix as code work; this file just captures the playtest-
finding lineage.

**Severity:** blocker for 14.11; nothing else in the live
playtest can run without it.

### S2 — `clientMode.ts` sends spectators with `playerID: null`

[../src/clientMode.ts:91-95](../src/clientMode.ts#L91-L95) — the
networked factory explicitly passes `playerID: null` for the
spectator path, which bgio routes as a non-seat connection. The
playerView code in [../src/game/playerView.ts](../src/game/playerView.ts)
strips secrets in that case (02.4). Looks correct.

### S3 — Server auth routes mounted before `Server.run()`

[../server/index.ts](../server/index.ts) calls
`mountAuthRoutes(server)` before `server.run`. Per bgio docs the
custom Koa router needs to be added to `server.router` before
boot — the order is right. Same for the idle watcher (10.9).

### S4 — In-flight auth body cap is 1 KiB

[../server/auth/middleware.ts](../server/auth/middleware.ts)
post-review-fix-#10. Should be enough for normal username+password
payloads (well under 256 B). If a real playtest hits a 413,
revisit.

### S5 — Client-side credential persistence is 24 h TTL

[../src/lobby/credentials.ts](../src/lobby/credentials.ts)
defines `CRED_TTL_MS`. Both tabs in the live run will need fresh
sessions if 24 h have passed since registration; otherwise reuse
should "just work."

### S6 — Chat hidden in hot-seat (14.7); should appear in networked

[../src/Board.tsx](../src/Board.tsx) gates chat on
`clientMode === 'networked'`. Live run will surface whether
`bgio` is actually delivering the chat messages — this is the
first session that should exercise that.

### S7 — Hot-seat seat picker is networked-mode read-only

[../src/ui/layout/SeatPicker.tsx](../src/ui/layout/SeatPicker.tsx) —
when `onChange` is undefined (no SeatPickerContext provider, the
networked-mode case in [../src/App.tsx](../src/App.tsx)) the
picker renders a read-only "You are Player N" badge. Confirm in
the live run that the lobby join sets the bgio Client's
`playerID` so the badge reads correctly.
