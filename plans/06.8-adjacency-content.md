# 06.8 — Adjacency content (initial pass)

**Parent:** [06-domestic.md](06-domestic.md)

## Goal

Author the first batch of adjacency rules so the registry from
[06.5](06.5-adjacency-rules.md) ships with real gameplay impact. Adjacency
is a V1 design lever, not flavor.

## Files

- Create `src/data/adjacency.json` — content.
- Create `src/data/adjacency.ts` — typed loader (mirrors 01.2 pattern).
- Modify `src/game/roles/domestic/adjacency.ts` to load from this file
  by default.
- Create `tests/data/adjacency.test.ts`.

## API

```ts
export interface AdjacencyRuleDef {
  defID: string;
  whenAdjacentTo: string | '*';
  bonus: Partial<ResourceBag>;
  flavor?: string;
}

export const ADJACENCY_RULES: ReadonlyArray<AdjacencyRuleDef>;
```

## Authoring approach (AI-assisted)

1. Feed an LLM the contents of [buildings.json](../src/data/buildings.json)
   and ask for ~20 thematically coherent rules:
   - Production buildings benefit from being next to resource buildings
     (Mill near Granary → +1 food).
   - Defense buildings benefit from clustering (Barracks next to Walls →
     attack +1).
   - Cultural buildings spread happiness (Fight Circle next to anything →
     +1 happiness).
2. Hand-curate the output for power-balance: no rule should yield more than
   doubling a building's solo output.
3. Encode the curated set in `adjacency.json`.
4. Loader validates: every `defID` and (non-`*`) `whenAdjacentTo` must
   match a name in `BUILDINGS`.

## Tests

- Loader rejects an unknown building id with a clear error.
- The registered rules + a fixture grid produce the expected bonus.
- Each `defID` appears in at most ~2 rules (sanity / power cap).
- 06.5's `yieldAdjacencyBonus` plus this content yields a non-empty bag
  for at least one buildable starter grid (so the feature is reachable
  early in a game).

## Dependencies

01.2, 06.5.

## Out of scope

- Negative adjacencies ("hospital next to factory: -1 food"). V1 stays
  positive-sum to keep the design space readable. Easy to add later.
