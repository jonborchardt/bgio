# 12.3 — Replay & move-log persistence

**Parent:** [12-testing-debug.md](12-testing-debug.md)

## Goal

Persist every move to `localStorage` (and optionally to disk in dev) so
real games are replayable bit-for-bit, simplifying bug repro.

**bgio-first stance:** the source of truth for "what moves happened"
is bgio's own `state.deltalog` / `state.log` — every action and the
RNG seed that produced it are already there. We **do not** intercept
move dispatch with a custom proxy; we read bgio's log directly and
serialize it. Replay drives a headless `Client` (`boardgame.io/client`)
through `client.store.dispatch(...)` of the same actions. Our code is
storage / UI; bgio is the recorder *and* the player.

## Files

- Create `src/replay/MoveLog.ts` — log type + storage.
- Create `src/replay/recorder.tsx` — React provider that subscribes to
  the bgio `Client` and appends moves.
- Create `src/replay/replay.tsx` — a route / mode that takes a log and
  drives the headless engine through it.
- Modify `src/App.tsx` to wrap with `<Recorder>` in dev.
- Create `tests/replay/replay.test.ts`.

## API

```ts
import type { LogEntry } from 'boardgame.io';

// MoveLog is a thin envelope around bgio's existing LogEntry[].
export interface MoveLog {
  matchID: string;
  numPlayers: 1|2|3|4;
  setupData: unknown;             // whatever was passed to lobby.createMatch
  entries: LogEntry[];            // bgio's type, NOT a redefined shape
}

export const snapshotLog: (client: BgioClient) => MoveLog;
// Reads client.log (or state.deltalog accumulated across states).

export const replay: (log: MoveLog) => SettlementState;
// Spins a headless Client (boardgame.io/client) with the same setupData;
// drives client.store.dispatch through log.entries; returns final G.

// Server endpoint: GET /games/settlement/<matchID>/log
//   → MoveLog (any logged-in user; matches are public per 10.3).
export const fetchLogFromServer: (
  serverUrl: string, matchID: string, authToken: string,
) => Promise<MoveLog>;
```

## Steps

1. `<Recorder>` subscribes to bgio's `Client` via `client.subscribe(...)`
   and reads `client.log` after each update. **No move-dispatch proxy.**
   The seed is part of bgio's RANDOM plugin state; we capture it via
   `setupData` and bgio's deterministic seeding (02.3) so replay is
   reproducible without us recording the seed separately.
2. Each updated log slice is written to `localStorage` under
   `settlement.replay.<matchID>`. Eviction policy: keep last 5 matches
   plus current.
3. `replay(log)`: spin a headless `Client` (`boardgame.io/client`) with
   the same `setupData`; iterate `log.entries` and dispatch each
   action through `client.store.dispatch`. bgio reconstructs identical
   `G` from its own log — we don't reimplement state transitions.
4. A `/replay?match=<id>` route resolves the log:
   - First check localStorage (instant load).
   - Else fetch via `fetchLogFromServer` (every match's bgio log
     lives in the storage adapter from [13.3](13.3-database-choice.md),
     so any logged-in user can replay any match).
   Then drive the engine; renders state at every step with prev/next
   controls.
5. **Server endpoint:** add `GET /games/settlement/:matchID/log` to
   `server/index.ts`. The handler reads `db.fetch(matchID, { log:
   true })` (bgio storage exposes the log) and returns `MoveLog`
   JSON. Auth: any valid token.

## Tests

- A 10-move recorded game replays to a state deep-equal to the live one.
- Eviction keeps exactly 5 matches.
- Replay route renders the correct state for step N.
- `fetchLogFromServer` returns the same log content as the local copy.
- Replay falls back to server when the localStorage entry is missing.
