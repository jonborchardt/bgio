# 08.5 — End conditions (win + 80-turn cap)

**Parent:** [08-cross-cutting.md](08-cross-cutting.md)

## Goal

Implement `endIf` for the `Settlement` game with two end states:

1. **Win** — `settlementsJoined >= 10`, per game-design.md §Goal.
2. **Time up** — `G.round >= TURN_CAP` (default 80, tunable per match).
   Not a "lose" — no punishment, no recovery mechanic. The run ends, the
   server records the score, and the player tries again to win faster.

Hitting the cap is the *only* non-win end. There is no creeping doom,
no health track, no bank-busted condition. The cap exists so the game
terminates and so RandomBot fuzz games end cleanly.

## Files

- Create `src/game/endConditions.ts`.
- Modify `src/game/index.ts` to set `endIf`.
- Modify `src/game/types.ts` to add `settlementsJoined: number` and
  `turnsAtWin?: number`.
- Create `tests/endConditions.test.ts`.

## API

```ts
export type GameOutcome =
  | { kind: 'win'; turns: number; settlementsJoined: number }
  | { kind: 'timeUp'; turns: number; settlementsJoined: number };

export const TURN_CAP_DEFAULT = 80;

export const endIf: (G: SettlementState, ctx: Ctx) => GameOutcome | undefined;
```

## Steps

1. `G.settlementsJoined` increments when:
   - Foreign wins a battle flagged `joins` (07.4).
   - Foreign successfully completes a tribute trade (07.5 follow-on).
2. `G.turnCap` set at `setup` from `setupData.turnCap ?? TURN_CAP_DEFAULT`.
   Tunable per match via the lobby form (10.3).
3. **Win check:** `settlementsJoined >= 10` →
   `{ kind: 'win', turns: G.round, settlementsJoined }`.
4. **Time-up check:** `G.round >= G.turnCap` →
   `{ kind: 'timeUp', turns: G.round, settlementsJoined }`.
5. Win takes precedence — if a player reaches 10 settlements on the same
   round the cap fires, they win.
6. **No other end states.**
   - Foreign battle losses apply tribute pressure (07.4) — slows you, never
     ends.
   - Empty bank, empty grid, etc. keep playing.
   - Wander deck (08.4) effects bite but never end.
7. **Score persistence:** the game-end hook (server-side, lives in
   [10.7-accounts.md](10.7-accounts.md)) reads the outcome and writes a
   `run` row to the DB keyed on the logged-in user. The win screen shows
   the user's personal best (turns to win) and best-so-far time-up
   (settlements joined at cap).

## Tests

- 9 settlements, round 50: `endIf` returns `undefined`.
- 10 settlements, round 50: `endIf` returns
  `{ kind: 'win', turns: 50, settlementsJoined: 10 }`.
- 0 settlements, round 80: `endIf` returns
  `{ kind: 'timeUp', turns: 80, settlementsJoined: 0 }`.
- 10 settlements **and** round 80: `endIf` returns `'win'` (precedence).
- `setupData.turnCap = 20` shortens the cap; cap fires at round 20.
- The persistence hook is called with the outcome (mock server in test).

## Dependencies

## Out of scope

- Lose conditions of any kind.
- Co-op vs competitive endings.
- Server-side leaderboards across users — personal-best only for V1.
  Public leaderboards become possible once accounts (10.7) are in.
