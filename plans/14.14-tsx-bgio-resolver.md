# 14.14 ‚Äî Fix `npm run server:dev` resolver vs bgio subpath imports

**Parent:** [14-playtest-followups.md](14-playtest-followups.md)

## Goal

Make `npm run server:dev` (and by extension `npm run dev:full`)
boot the bgio Koa server cleanly. Today it dies on first import
because tsx 4.20's bare-specifier resolver appends `.jsx` to
subpath imports like `boardgame.io/server` instead of reading the
target directory's `package.json` `main` field. See
[14.11-networked-playtest-findings.md F1](14.11-networked-playtest-findings.md)
for the full diagnosis.

## Why

This blocks 14.11 (the live two-tab networked playtest) and any
local dev workflow that runs the server. Production deploys are
unaffected (the Dockerfile builds to JS and runs Node directly),
and tests are unaffected (Vitest uses Vite's resolver).

The findings session probed three workarounds; only fixing the
runner itself is sustainable, since 27 files in `src/` import
from bgio subpaths.

## Files

- Modify [`../package.json`](../package.json) ‚Äî change the
  `server:dev` / `server:start` scripts (and `dev:server` /
  `dev:full`).
- Possibly add a devDependency (`vite-node`) ‚Äî if option 2 below
  is taken.
- Update [`../server/README.md`](../server/README.md) and
  [`../README.md`](../README.md) if the dev command name
  changes.

## Steps

Pick **one** of the candidate fixes; verify with the validation
gate below.

### Candidate A ‚Äî pin tsx to a known-good version

1. `npm install --save-dev tsx@4.7.3` (last release before the
   `resolveExtensions` rework ‚Äî verify by reading tsx's
   CHANGELOG and bisecting if needed).
2. Run `npm run server:dev`; confirm the server prints
   `App serving on 8000...`.
3. Run `npm test`; confirm the boot smoke + every other test
   still passes.
4. If green, commit the package.json + lockfile change.

Risk: tsx might have shipped Node 20+ or Windows fixes after the
known-good version that we lose by pinning.

### Candidate B ‚Äî switch to vite-node

1. `npm install --save-dev vite-node`.
2. Replace the `server:dev` / `server:start` scripts with
   `vite-node server/index.ts`. Drop the `--watch` flag; vite-node
   has its own watch mode (`--watch`). Update `dev:server` to
   `vite-node --watch server/index.ts`.
3. Verify the server boots and the boot smoke test still passes
   (it uses vitest's resolver, which is the same family as
   vite-node ‚Äî likely no change needed).
4. If vite-node bundles less aggressively than tsx and surfaces
   any per-import behavior differences, fix those forward.

Risk: extra dev dep, but tsx becomes a per-file ad-hoc tool
instead of a critical-path runner.

### Candidate C ‚Äî build then run with Node

1. Update `server:dev` to `tsc -p server && node dist-server/server/index.js`
   (assuming a tsconfig that emits to `dist-server/`).
2. Add a watch loop via `tsc -w` + a process supervisor.

Risk: slower iteration; loses tsx's transparent TS handling. Use
only if A and B both blow up.

## Tests

- `npm run server:dev` boots without errors and the bgio Koa
  server serves on the configured port (default 8000).
- The 14.11 live playtest can now run end-to-end.
- `npm test` still green (482+ tests).
- `npm run typecheck && npm run lint` clean.

## Dependencies

10.1 (server module), 12.7 (`dev:full`), 14.11 (the playtest
that surfaced this).

## Out of scope

- Reworking the entire server's import style. The bgio subpath
  imports are correct; the runner is at fault.
- Filing the upstream tsx bug. Worth doing but not required to
  unblock the local dev loop.
