# 10.5 — Chat

**Parent:** [10-multiplayer.md](10-multiplayer.md)

## Goal

In-match game-chat between players. **Public only — no per-team or
per-role private channels in V1.**

**bgio-first stance:** bgio 0.50.x ships a built-in chat channel —
the `Client` exposes `chatMessages: ChatMessage[]` and a
`sendChatMessage(text)` method, and the Server transports + persists
them automatically. We **do not** write a custom plugin or move for
chat; we only write the React surface that reads / writes through the
existing API. If we ever need length / content limits, those go in
the composer (client-side guard), with optional reinforcement via
bgio's `authenticateCredentials` hook server-side.

## Files

- Create `src/ui/chat/ChatPane.tsx` — reads `chatMessages` off the bgio
  Client.
- Create `src/ui/chat/ChatComposer.tsx` — calls
  `client.sendChatMessage(payload)`.
- Modify `src/Board.tsx` to render `<ChatPane>`.
- Create `tests/ui/chat/ChatPane.test.tsx`,
  `tests/ui/chat/ChatComposer.test.tsx`.

## API

```ts
// No new game-side API. We consume bgio's built-ins:
//
//   client.chatMessages: ChatMessage[]   // { id, sender: PlayerID, payload: any }
//   client.sendChatMessage(payload)      // payload: anything serializable
//
// We define a small payload shape so panes can render consistently:
export interface ChatPayload {
  text: string;        // length-clamped 1..280 by composer
  ts: number;          // client-stamped; server timestamp lives on the bgio envelope
}
```

## Steps

1. `<ChatComposer>` calls `client.sendChatMessage({ text, ts: Date.now() })`.
   - Length-clamps to 280 chars; trims; rejects empty.
   - HTML is escaped at render time (React already does this; never
     `dangerouslySetInnerHTML`).
2. `<ChatPane>` subscribes to bgio's `Client` and re-renders on
   `chatMessages` changes — the bgio React `Client` factory exposes
   `chatMessages` as a prop on the wrapped board, so the pane can
   read it from `props` directly.
3. Auto-scroll to bottom on new message.
4. Persistence: bgio's chat is part of the match transport layer and
   stored alongside metadata by the storage adapter (10.4). No
   plugin registration, no game-state slice, no custom move.
5. **Spectators** (10.8) see the same chat — `sendChatMessage` is
   gated by bgio to authenticated participants; spectators read but
   can't post (matches our V1 spec).

## Tests

- `ChatComposer` calls `client.sendChatMessage` with the trimmed,
  length-clamped payload.
- Empty / >280-char text → composer disables the send button (no
  network call).
- HTML in `text` renders as text, not markup (React default).
- ChatPane scrolls to bottom when `chatMessages` grows.
- An integration test (two headless clients via `Local` transport)
  confirms a message sent from client A appears in client B's
  `chatMessages` — this exercises bgio's transport, not our code.

## Dependencies

10.1 (bgio Server transports chat), 10.2 (Client wired with chat
support — networked transport already includes it).

## Out of scope

- Per-team / per-role private chat. Reserved for later.
- Chat moderation / muting.
