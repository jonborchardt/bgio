# 11.2 — MCTS baseline

**Parent:** [11-ai.md](11-ai.md)

## Goal

Wire bgio's `MCTSBot` and define `enumerate` so MCTS can list legal moves.
This both gives us a stronger-than-random baseline and pressures the move
surface toward enumerability — a property we want anyway.

## Files

- Create `src/game/ai/enumerate.ts`.
- Modify `src/game/index.ts` to attach `ai: { enumerate }`.
- Create `tests/ai/enumerate.test.ts`.
- Create `tests/ai/mcts.smoke.test.ts`.

## API

```ts
import type { Ctx } from 'boardgame.io';
import type { SettlementState } from '../types';

export interface MoveCandidate { move: string; args: unknown[]; }

export const enumerate: (
  G: SettlementState, ctx: Ctx, playerID: string,
) => MoveCandidate[];
```

## Steps

1. `enumerate` examines `ctx.phase` + `ctx.activePlayers?.[playerID]` and
   returns the legal moves available to `playerID`:
   - In `chiefPhase`: `chiefDistribute` choices for each (target seat,
     resource subset), `chiefEndPhase`, optionally `chiefPlaceWorker` /
     `chiefPlayGoldEvent`.
   - In `othersPhase` `scienceTurn`: `scienceContribute` and
     `scienceComplete` candidates.
   - Etc.
2. Combinatoric blow-up control: cap each per-resource step at small
   integers (e.g. distribute in increments of 1). MCTS uses these
   candidates as branching points.
3. The enumerator does not validate full legality — it relies on the move
   itself returning `INVALID_MOVE` on any bad combo. Goal is to enumerate
   "plausible" candidates so MCTS doesn't waste time.

## Tests

- `enumerate` in `chiefPhase` returns exactly the chief moves.
- `enumerate` in `scienceTurn` does NOT return chief moves.
- A 5-step `MCTSBot` smoke test runs without throwing for `numPlayers=2`.
