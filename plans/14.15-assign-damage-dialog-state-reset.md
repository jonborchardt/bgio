# 14.15 — AssignDamageDialog wipes user edits on re-render

**Parent:** [14-playtest-followups.md](14-playtest-followups.md)

## Goal

Stop `AssignDamageDialog` (14.10) from blowing away the user's
edits whenever the bgio Client re-renders the Board.

## Why

[`src/ui/foreign/AssignDamageDialog.tsx`](../src/ui/foreign/AssignDamageDialog.tsx)
recomputes the per-event default allocations in a
`useEffect([events, playerRows])`. The `events` `useMemo`
depends on `inPlay`, which is a fresh `UnitInstance[]` array
every time bgio's Client re-renders the Board (BoardProps state
is recreated each tick). So:

1. User opens the dialog.
2. User clicks `+1` on Brute — local state updates.
3. Anything triggers a Board re-render (chat message, idle tick,
   another seat's move): `inPlay` is a new array, `events` is a
   new array, `useEffect` fires, `setAllocations` resets the
   user's edits to the greedy default.

The dialog is effectively unusable for any allocation that
deviates from greedy.

## Files

- [`src/ui/foreign/AssignDamageDialog.tsx`](../src/ui/foreign/AssignDamageDialog.tsx)

## Steps

1. Reset `allocations` only when the dialog **opens** (`open`
   transitions `false → true`) OR when the **shape** of `events`
   changes (per-event `incoming` numbers differ — different
   battle).
2. Memoize `events` against a stable key (`battle.id` +
   stringified `inPlay` summary) so identity stays the same
   across cosmetic re-renders.
3. Or simpler: track `lastBattleId` in state; reset allocations
   only when `inFlight.battle?.id` changes from the previous
   render.

Approach 3 is the smallest change.
