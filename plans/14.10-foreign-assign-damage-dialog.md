# 14.10 — Foreign AssignDamageDialog: real per-round absorber UI

**Parent:** [14-playtest-followups.md](14-playtest-followups.md)

## Goal

The 07.3 battle resolver consumes one `DamageAllocation` per
incoming-damage event from enemy → player. The current
[../src/ui/foreign/AssignDamageDialog.tsx](../src/ui/foreign/AssignDamageDialog.tsx)
ships a one-shot stub that emits
`[{ byUnit: { [firstDefID]: 1 } }]` regardless of the battle's
shape, so the move INVALID_MOVEs on any battle that produces more
than one incoming hit — see playtest issue E2.

## Files

- Modify `src/ui/foreign/AssignDamageDialog.tsx`.
- Optionally extract `src/ui/foreign/allocator.ts` for the
  per-round preview math.
- Modify `tests/ui/foreign/ForeignPanel.test.tsx`.

## Steps

1. Build a "preview" pass: drive `resolveBattle` with the player +
   enemy units and the *empty* allocations array; capture the
   resolver's mid-state at each enemy → player damage event so we
   know how many allocations the dialog needs to ask the player
   for.
2. For each incoming-damage event, render a chip strip "Incoming
   N damage" with a per-unit slider / stepper letting the player
   distribute the damage across their committed units. Live-sum
   under the strip; disable submit until every strip's sum equals
   the incoming amount.
3. On submit, build `DamageAllocation[]` (one per strip) and
   call `props.moves.foreignAssignDamage(allocations)`.
4. Greedy default: pre-fill each strip with "lowest-HP unit
   absorbs first" so a single click submits a sane plan.
5. Cancel button discards the dialog and leaves the seat in
   `foreignAwaitingDamage` for retry.

## Tests

- A 2-incoming-event battle yields 2 strips.
- The submit button enables only when every strip's sum equals its
  incoming amount.
- Greedy default yields a submit-ready plan.
- Cancel leaves G unchanged.

## Dependencies

07.3 (resolveBattle), 07.4 (foreignAssignDamage move), 07.7
(ForeignPanel host).

## Out of scope

- Animations of the resolver step-through.
- A "what-if" preview that re-runs the resolver against
  partially-edited allocations. That's a richer interaction; V1
  ships the deterministic submit gate above.
