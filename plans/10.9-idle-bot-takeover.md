# 10.9 — Idle bot takeover

**Parent:** [10-multiplayer.md](10-multiplayer.md)

## Goal

A multiplayer match doesn't stall when a player logs out, closes the tab,
or just stops acting. After `IDLE_TIMEOUT` (default 5 minutes) of no input
on the seat's expected stage, a server-side bot takes over the seat for
the rest of the match. The human can reclaim the seat by reconnecting.

**bgio-first stance:** the bot itself is configured via bgio's existing
`bots` slot on `Server({ games, db, bots })` and runs through bgio's
internal bot driver (the same one stage 11 wires for `RandomBot` /
`MCTSBot` / heuristic bots). We only add the *idle-watching* layer —
which seat is currently considered bot-controlled — and patch
`match.metadata.bots` accordingly. We do not write a parallel bot
runtime.

## Files

- Create `server/idle/idleWatcher.ts` — per-match watcher.
- Create `server/idle/seatTakeover.ts` — swap a seat from human to bot
  and back.
- Modify `server/index.ts` to start watchers per active match.
- Create `tests/server/idle.test.ts`.

## API

```ts
export const IDLE_TIMEOUT_MS = 5 * 60 * 1000;

export interface IdleWatcher {
  start: () => void;
  stop: () => void;
  noteActivity: (matchID: string, playerID: PlayerID) => void;
}

export const makeIdleWatcher: (server: BgioServer) => IdleWatcher;

export const grantBotControl: (matchID: string, playerID: PlayerID) => Promise<void>;
export const revokeBotControl: (matchID: string, playerID: PlayerID) => Promise<void>;
```

## Steps

1. The watcher records `lastActivity[matchID][playerID] = now()` on every
   inbound move and on every connect ping.
2. A periodic sweep (every 30s) checks each active match: if a seat's
   stage is "their turn" AND `now - lastActivity > IDLE_TIMEOUT_MS`, call
   `grantBotControl`.
3. `grantBotControl`:
   - Adds the seat to `match.metadata.bots` (bgio's standard slot —
     this is what `Server` reads to decide which seats it auto-drives).
   - Lets bgio's built-in bot driver handle the actual moves; the
     per-role heuristic from 11.x is registered on the server's
     `bots` config keyed by role, so the driver dispatches correctly.
   - Posts a chat message via `client.sendChatMessage` (10.5):
     `"Bot took over for <username> (idle)"`.
4. On reconnect: `revokeBotControl` removes the seat from
   `match.metadata.bots` (bgio stops driving it automatically) and
   posts `"<username> reclaimed their seat"`.
5. The watcher is bounded (one timer per server, not per match) to keep
   memory linear in match count.

## Tests

- A 2p match: human in seat 1 stops acting on their stage; after the
  timeout the bot moves the game forward.
- Reconnect: bot stops, human resumes from current state.
- A connection that's still active but on a stage other than its own
  (waiting on chief) does NOT trigger takeover.
- Watcher survives a single bad bot move (logged, not crash-the-watcher).

## Dependencies

10.1, 10.7 (accounts so we know whose seat it was), 11.3–11.6 (the bots
themselves).

## Out of scope

- Tunable per-match timeout. V1 ships one constant.
- "Pause match" alternative — would freeze the game until the human
  returns; rejected because long pauses kill momentum for the other
  players.
