# 08.2 — Event-effect dispatcher

**Parent:** [08-cross-cutting.md](08-cross-cutting.md)

## Goal

A typed registry of event effects. Effects are data; the dispatcher applies
them at well-defined hook points across the engine. New event content adds
data, never engine code, unless a new effect kind is genuinely required.

## Files

- Create `src/game/events/effects.ts`.
- Create `src/game/events/dispatcher.ts`.
- Create `src/game/events/hooks.ts` — exported "where to call dispatch"
  helpers used by other modules.
- Create `tests/events/dispatcher.test.ts`.

## API

```ts
export type EventEffect =
  | { kind: 'gainResource'; bag: Partial<ResourceBag>; target: 'bank' | 'wallet' }
  | { kind: 'doubleScience' }                  // sci pays 2x advancement this turn
  | { kind: 'forbidBuy' }                       // sci can't complete a card this turn
  | { kind: 'forceCheapestScience' }
  | { kind: 'swapTwoScienceCards' }            // requires user pick → use 'awaitInput' below
  | { kind: 'redrawBattleTop' }
  | { kind: 'tributeWaiver' }
  | { kind: 'addEventCard'; cardID: string }    // appends to a color deck
  | { kind: 'awaitInput'; prompt: string; payloadKind: string };

export const dispatch: (
  G: SettlementState, ctx: Ctx, random: RandomAPI,
  card: EventCardDef, payload?: unknown,
) => void;

// Other modules call these to evaluate active effect mods.
export const hasModifierActive: (G: SettlementState, kind: EventEffect['kind']) => boolean;
export const consumeModifier: (G: SettlementState, kind: EventEffect['kind']) => void;
```

## Steps

1. Effects with deterministic outcomes apply immediately
   (`gainResource`, `addEventCard`, `redrawBattleTop`, `tributeWaiver`).
2. Effects that **modify subsequent moves** push a "modifier" entry onto
   `G._modifiers: EventEffect[]`. Producers of those moves
   (`scienceContribute`, `domesticBuyBuilding`, etc.) check
   `hasModifierActive(...)` and call `consumeModifier(...)` after applying.
3. Effects requiring user input (`swapTwoScienceCards`) use the
   `awaitInput` mechanism: enter the `playingEvent` stage from 02.2 with a
   payload schema; a follow-up move (`eventResolve(payload)`) feeds the
   payload back to the dispatcher.
4. Dispatcher throws on an unknown `kind` so we notice content drift.

## Tests

- `gainResource` adds to the targeted bag.
- `doubleScience` modifier doubles a single subsequent
  `scienceContribute` and is then consumed.
- `forbidBuy` blocks `scienceComplete` until end of round.
- `swapTwoScienceCards` enters `playingEvent`; follow-up `eventResolve`
  swaps the two indicated cards in the grid.
- Unknown effect kind → throws.

## Dependencies

01, 02, 08.1, plus the role moves it modifies (so the modifier consumption
sites exist). Each role move adds a single line consulting the dispatcher;
that's their contract with this module.

## Out of scope

- Designing the actual content. We list the effect kinds; data fills in.
