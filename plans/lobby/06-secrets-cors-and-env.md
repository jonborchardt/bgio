# 06 — Secrets, CORS, and env wiring on Cloud Run

**Severity**: medium (small but easy to get wrong; will silently break the lobby)
**Area**: server / config
**Effort**: small
**Status**: not started
**Depends on**: plans 02 + 05 (we need both URLs)

## Files
- [server/index.ts:89-99](../../server/index.ts) — reads `ALLOWED_ORIGINS`
  and forwards it to bgio's SocketIO transport.
- [server/auth/routes.ts:32-54](../../server/auth/routes.ts) — same env,
  used to set the `Access-Control-Allow-Origin` header on `/auth/*` routes.
- [server/index.ts:121-129](../../server/index.ts) — `TRUST_PROXY` gates
  `koaApp.proxy = true`.
- [cloudrun/service.yaml](02-cloud-run-deploy.md) — env-var assignment.
- [.env.example](../../.env.example) — already documents these.

## Problem

Two independent things need to line up between the server and Pages
build, and neither is checked by tests:

1. **`ALLOWED_ORIGINS` on the server must include the Pages URL.** If
   it doesn't, the browser silently refuses every cross-origin auth
   request and SocketIO upgrade. The user sees "request failed: 0" in
   `<AuthForms>` and a stuck "Loading…" in the lobby.
2. **`VITE_SERVER_URL` baked into the Pages bundle must point at the
   Cloud Run URL.** If it points at localhost, the bundle works only
   when the user happens to be running a local server.

There is no runtime probe that catches this; it only fails when a real
browser tries to log in.

## Fix sketch

### 1. `ALLOWED_ORIGINS` is non-secret; commit it

The Pages URL is publicly visible — there is no value in secret-storing
it. Set it as a plain env var on the Cloud Run service:

```yaml
env:
  - name: ALLOWED_ORIGINS
    value: https://YOUR-ORG.github.io
```

If we end up running multiple Pages URLs (preview branches, custom
domain), set the comma-separated list:

```yaml
  - name: ALLOWED_ORIGINS
    value: https://YOUR-ORG.github.io,https://settlement.example.com
```

The server's split-on-comma is already in place at
[server/index.ts:93](../../server/index.ts).

### 2. `TRUST_PROXY=true`

Cloud Run terminates TLS at its edge and the application sees
`X-Forwarded-For`. Set this so the auth rate-limit buckets at
[server/auth/routes.ts:228](../../server/auth/routes.ts) bucket by
real client IP, not by the proxy's address.

```yaml
  - name: TRUST_PROXY
    value: "true"
```

### 3. There are currently no actual secrets

Audit:

- `STORAGE_KIND`, `SQLITE_PATH`, `PORT`, `NODE_ENV` — non-secret,
  commit.
- `ALLOWED_ORIGINS`, `TRUST_PROXY` — non-secret, commit.
- Auth tokens generated by [server/auth/accounts.ts] are random
  strings persisted in the same SQLite DB — no signing key, no
  JWT, nothing to inject.
- Password hashes use scrypt with a per-user salt (random) — no
  shared pepper, no secret to rotate.

If we ever add HMAC-signed tokens or third-party OAuth, the right
home is **Google Secret Manager** + `--set-secrets` on the Cloud Run
service:

```yaml
  - name: AUTH_SIGNING_KEY
    valueFrom:
      secretKeyRef:
        name: settlement-auth-signing-key
        key: latest
```

But at V1, **there are no secrets to wire**. Don't preemptively add
Secret Manager — the cost is real and unused.

### 4. Workload Identity Federation for the deploy CI

The CI workflow in plan 02 authenticates to GCP via WIF (no JSON key
in repo secrets). The two repo-level secrets we need are non-secret-ish:

- `GCP_WIF_PROVIDER` — the provider resource path (looks like
  `projects/123/locations/global/workloadIdentityPools/github/providers/github`).
  Public information for our setup but treated as a secret per
  Google's recommendation.
- `GCP_DEPLOY_SA` — the service account email
  (e.g. `cloud-run-deployer@PROJECT.iam.gserviceaccount.com`).

These go in the GitHub repo settings → Secrets and variables →
Actions, scoped to the `deploy-cloudrun` workflow.

### 5. Pages build env var

`VITE_SERVER_URL` is **not** secret — it ends up in the bundle
verbatim. Use a repo **variable** (not secret):

- Repo settings → Variables → `VITE_SERVER_URL` =
  `https://settlement-server-<hash>-uc.a.run.app`
- Plan 05 already references `${{ vars.VITE_SERVER_URL }}` in the
  workflow snippet.

## Acceptance

- `gcloud run services describe settlement-server --region=us-central1
  --format='value(spec.template.spec.containers[0].env)'` shows
  `ALLOWED_ORIGINS` and `TRUST_PROXY=true` set.
- The Pages build's bundle (under `dist/assets/`) contains the Cloud
  Run URL string, not `localhost`.
- Hitting the Pages URL → DevTools Network panel:
  - `OPTIONS https://...run.app/auth/login` returns 204 with
    `access-control-allow-origin: https://YOUR-ORG.github.io`.
  - `POST` after that succeeds.
- `gcloud run services logs read settlement-server` shows real client
  IPs for auth attempts (not all `35.x.x.x` from Google's edge).

## Risks / open questions

- **Origin mismatch is silent.** If `ALLOWED_ORIGINS` and the Pages
  URL drift the lobby breaks. Add a check to plan 07 acceptance that
  hits `/auth/verify` from the live Pages URL.
- **Forgetting `TRUST_PROXY=true`.** Without it the rate limiter
  collapses every attacker into one bucket (issue 023's old bug).
  The server log line `[auth] all requests bucket as <IP>` is the
  tell — verify in plan 07.
- **Custom domain later.** Adding a custom domain to Cloud Run would
  add a new origin that *also* needs to be in `ALLOWED_ORIGINS`. Just
  remember to update both the env and the Pages build.
