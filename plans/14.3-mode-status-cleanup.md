# 14.3 — Mode label + CenterMat dedup

**Parent:** [14-playtest-followups.md](14-playtest-followups.md)

## Goal

Two small UX wins:

1. StatusBar "Mode" reflects the actual client mode
   (`hot-seat` / `networked` / `spectating`), not the redacted
   spectator detection. Today hot-seat displays "Spectating" because
   `playerID === undefined`. (Issue B3.)
2. The standalone `<CenterMat>` row duplicates per-seat info that
   `<ChiefPanel>`'s CircleEditors already render. Hide the
   standalone row when ChiefPanel is visible. (Issue V3.)

## Files

- Modify `src/clientMode.ts` — export the resolved mode as a const
  callers can read at render time.
- Modify `src/ui/layout/StatusBar.tsx` — render the new mode label.
- Modify `src/Board.tsx` — pass the mode through; conditionally
  render `<CenterMat>`.

## Steps

1. `getClientMode()` (extending `detectMode`) returns `'hotseat' |
   'networked' | 'spectating'`. Spectating is when `playerID === null`
   in networked mode (10.8 spectator flow). Hot-seat is when
   `clientMode === 'hotseat'`. Networked otherwise.
2. StatusBar takes a `mode` prop and renders it (capitalized:
   "Hot-seat", "Networked", "Spectating").
3. Board.tsx renders `<CenterMat>` only when the local viewer does
   NOT hold chief. The ChiefPanel CircleEditors are sufficient for
   the chief's view; non-chief seats need the read-only mat.

## Tests

- Hot-seat default → mode shows "Hot-seat".
- Networked mode with playerID set → "Networked".
- Networked mode with playerID === null → "Spectating".
- Chief seat: <CenterMat> not in DOM. Non-chief seat: present.

## Dependencies

09.1 (StatusBar), 09.3 (CenterMat), 10.2 (clientMode), 04.5
(ChiefPanel), 14.1 (seat picker — required so we can flip viewer
to test the chief / non-chief render branches).

## Out of scope

- Merging CenterMat and ChiefPanel into a single composite
  widget. Keeping them separate so role panels can render the mat
  view standalone in different layouts later.
