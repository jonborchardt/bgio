# 13.3 — Database choice

**Parent:** [13-deploy.md](13-deploy.md)

## Goal

Pick the production database and wire its `Storage` adapter.

## Decision

**SQLite via `better-sqlite3`** for V1. Reasons:
- Single match per row, modest write volume — SQLite handles it trivially.
- Both Render and Fly support a persistent disk; SQLite needs nothing else.
- No connection pool to maintain.
- Migration to Postgres later is a non-event because everything goes
  through the `Storage` interface from 10.4.

## Files

- Promote `server/storage/sqlite.ts` from "stub" to a real implementation.
- Create `server/storage/migrations/` — versioned SQL.
- Modify `server/index.ts` to default to `kind: 'sqlite'` when
  `STORAGE_KIND` env is unset.
- Create `tests/server/sqlite.test.ts`.

## Steps

1. Schema (bgio match state + chat):
   ```sql
   CREATE TABLE IF NOT EXISTS matches (
     id TEXT PRIMARY KEY,
     setup_data TEXT NOT NULL,
     metadata TEXT NOT NULL,
     state TEXT NOT NULL,
     created_at INTEGER NOT NULL,
     updated_at INTEGER NOT NULL
   );
   CREATE TABLE IF NOT EXISTS log (
     match_id TEXT NOT NULL,
     idx INTEGER NOT NULL,
     entry TEXT NOT NULL,
     PRIMARY KEY (match_id, idx),
     FOREIGN KEY (match_id) REFERENCES matches(id)
   );
   ```
   Account / auth / run-history schemas live in
   [10.7](10.7-accounts.md) and are added by their own migration files
   in the same `migrations/` directory.
2. Migrations are idempotent SQL files run at boot in numbered order.
3. Implement bgio's `Async` storage interface against the schema, with
   `JSON.stringify` / `JSON.parse` for the payload columns.
4. WAL mode for concurrency.

## Tests

- Round-trip a match through the real adapter (in-memory `:memory:`).
- `state` column survives a process restart (file-backed test).
- Concurrent writes from two clients don't corrupt — bgio serializes per
  match anyway, but we verify the storage doesn't deadlock.

## Dependencies

10.4 (interface), 13.2 (host with persistent disk).

## Out of scope

- Cross-region replication.
- Postgres adapter — written only when SQLite is the bottleneck, with
  a documented threshold (e.g. >10 concurrent matches sustained).
