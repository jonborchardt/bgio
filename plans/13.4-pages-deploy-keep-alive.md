# 13.4 — GitHub Pages deploy keep-alive

**Parent:** [13-deploy.md](13-deploy.md)

## Goal

Make sure the existing GitHub Pages workflow at
[.github/workflows/deploy-pages.yml](../.github/workflows/deploy-pages.yml)
keeps working through every later stage. The Pages build is the
**ephemeral demo path** — no login, no run recording, no persistence.
It exists so a curious visitor can see the game without us paying for
a server seat. The canonical experience is the networked Render deploy.

## Files

- Modify `.github/workflows/deploy-pages.yml` — small additions only
  (concurrency group, ensure `build:hotseat` is the script invoked).
- Modify `12.4` CI workflow to run `npm run build:hotseat` so PRs catch
  Pages-breaking changes before merge.

## Steps

1. Pin Node to `20.x` for repeatability.
2. `concurrency: { group: pages, cancel-in-progress: true }` so a fast
   second push supersedes the first deploy.
3. Build script: `npm run build:hotseat` (post-13.1 default).
4. Verify `dist/` size stays under GH Pages' soft 1GB site limit; assets
   from 09.x (cards, theme images) shouldn't push us close.
5. README mention: "Demo (no save, no login) at the GitHub Pages URL.
   Full experience with accounts and run history requires the networked
   server (see server/README.md)."
6. The Pages build must hide the "log in" / "register" UI and any
   "personal best" affordances — those depend on the server. Gate via
   the `VITE_CLIENT_MODE` from 10.2.

## Tests

- A push to a branch with a Pages-breaking change fails CI before being
  merged.
- A push to `main` deploys; the resulting site loads in a browser and
  the smoke spec passes against the deployed URL.

## Dependencies

10.2, 13.1, 12.4, 12.5.

## Out of scope

- Custom domain configuration.
- Versioned deploys with rollback.
