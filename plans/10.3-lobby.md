# 10.3 — Lobby

**Parent:** [10-multiplayer.md](10-multiplayer.md)

## Goal

Host can create a match (1-4 players); joiners pick a seat and bind to a
role. Provides the `matchID` / `playerID` / `credentials` triple to 10.2.

**bgio-first stance:** use `Lobby` from `boardgame.io/react` for match
listing / creation / join / leave / play-again, and `LobbyClient` from
`boardgame.io/client` for any headless REST calls. Do **not** hand-roll
fetch wrappers around `/games/:name/create`, `:matchID/join`,
`:matchID/leave`, `:matchID/playAgain` — `LobbyClient` already wraps every
endpoint. We only build UI on top.

## Files

- Create `src/lobby/LobbyShell.tsx` — thin wrapper that renders bgio's
  `Lobby` component, supplying our `gameComponents`, server URL, render
  overrides, and auth-gated entry.
- Create `src/lobby/SeatPicker.tsx` — passed to `Lobby` as a render
  override (the part bgio's stock UI doesn't know about: which seat
  maps to which role per `assignRoles(numPlayers)`).
- Create `src/lobby/lobbyClient.ts` — single shared `new LobbyClient({
  server })` instance for any non-`Lobby` flows (server-down probes,
  spectator listing for 10.8, idle-takeover RPC stubs).
- Modify `src/main.tsx` to mount `<LobbyShell>` when `clientMode ===
  'networked'` and no match has been picked yet.
- Create `tests/lobby/lobbyClient.test.ts`.

## Steps

1. `lobbyClient.ts` constructs one `LobbyClient` pointing at
   `VITE_SERVER_URL`. All non-`Lobby` callers (e.g., 10.6 reconnect
   probe, 10.8 spectator list refresh, 10.9 takeover RPCs) consume
   this shared client. **Don't write a sibling `api.ts` that re-wraps
   the same endpoints.**
2. `LobbyShell` renders bgio's `Lobby` component:
   - Gate on logged-in user (10.7); render `<AuthForms>` until login,
     then enter the lobby with `playerName` = account username.
   - bgio's `Lobby` already drives list/create/join/leave/play-again —
     we just provide a custom `renderer` to MUI-ify it and inject the
     seat-with-roles picker.
3. `SeatPicker` is the role-binding UI bgio's stock lobby doesn't know
   about. It receives `numPlayers` + the room's current seat occupancy
   (from `Lobby`'s renderer args) and labels each seat with the
   role(s) per `assignRoles(numPlayers)`. Seat selection calls
   `handleJoinRoom(playerID)` — bgio still owns the join.
4. Tunable knobs (starting bank, turn cap default 80, solo toggle for
   11.7) ride on `setupData` passed to `lobby.createMatch()`. Game
   `setup` reads it. **Do not invent a parallel match-config store.**
5. Auth: on join, `Lobby` returns `playerCredentials`. Store
   `(matchID, playerID, credentials, authToken)` in `localStorage`
   and pass to the game `Client`. Player name comes from the
   logged-in account; no separate name field in the lobby.
6. "Watch" → spectator flow (10.8) is a `joinMatch(playerID = null)`
   on `LobbyClient` — bgio supports it natively.

## Tests

- `lobbyClient` exposes the bgio methods (smoke test: shape of the
  exported instance, no need to re-test bgio's REST wiring).
- `LobbyShell` renderer maps `numPlayers ∈ 1..4` to the right
  seat-with-roles grid.
- Selecting a seat calls `handleJoinRoom` with the matching `playerID`.
- `setupData` round-trips: a match created with custom turn cap reads
  back the value via `Lobby`'s match list.
