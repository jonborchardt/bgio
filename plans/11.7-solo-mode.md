# 11.7 — Solo mode

**Parent:** [11-ai.md](11-ai.md)

## Goal

A "Solo" lobby option where one human plays one role and bots cover the
others. Wires 11.3–11.6 via bgio's `MatchData.bots` config.

## Files

- Modify `src/lobby/CreateMatchForm.tsx` — add Solo mode toggle.
- Create `src/lobby/soloConfig.ts` — builds the bot map.
- Modify `src/App.tsx` to attach bots to the local `Client` (or, in
  networked mode, to spin server-side `runBot` workers — V1 keeps it
  client-side).
- Create `tests/lobby/soloConfig.test.ts`.

## API

```ts
export interface SoloConfig {
  numPlayers: 1 | 2 | 3 | 4;
  humanRole: Role;     // which role the human plays
}

export const buildBotMap: (cfg: SoloConfig) => Record<PlayerID, BotImpl>;
```

## Steps

1. **Default solo config: `numPlayers: 2`.** The human picks one of the
   two valid 2p seat-pairs (per game-design.md §Players):
   - **Chief + Science** (seat 0), bot plays Domestic + Foreign (seat 1).
   - **Domestic + Foreign** (seat 1), bot plays Chief + Science (seat 0).
   This matches the user-stated solo experience: "like 2-player, but with
   bots." It plays faster than 4-player and exercises the role-doubling
   path that 1p/2p configs use.
2. Solo also supports `numPlayers ∈ {1,3,4}` for variety:
   - 1p: human owns all four roles, no bots.
   - 3p / 4p: human picks one role; bots cover the rest.

   In 3p, only **Chief+Science** is a doubled seat — so if humanRole is
   chief or science, the bot covers Domestic and Foreign individually;
   if humanRole is domestic or foreign, the bot's seat 0 plays the
   chief+science pair.
3. `buildBotMap` resolves `humanRole` to a seat, then assigns the per-role
   bot to every other seat. For doubled roles (1p/2p/3p), the bot for the
   seat is a composed bot that runs each owned role in turn.
4. UI: a "Solo" toggle in `CreateMatchForm` reveals the numPlayers picker
   and the role picker; on submit, store config in match metadata so
   reload restores correctly.
5. **Solo runs through the server, not local.** The user explicitly
   chose to persist single-player games to the server SQLite DB via the
   accounts system (10.7). The match is created normally; bots are
   driven by server-side `runBot` workers in the bgio server process —
   the same path used by [10.9](10.9-idle-bot-takeover.md), so there is
   exactly one bot infrastructure to maintain.
   This means the GH Pages hot-seat fallback is *demo-only*, with no
   persistence — a single-player run that the user wants to remember
   requires login + server.
6. **Dev loop.** Local dev runs `npm run server:dev` + `npm run dev` in
   two terminals. The lobby points at `http://localhost:8000` via
   `VITE_SERVER_URL`. A "Solo" match created from the local lobby spins
   bots inside the local server — same architecture as production, just
   on localhost. No special "client-only solo" toggle exists.

## Tests

- **Default 2-player solo:** seat 0 is human (holds Chief+Science by
  default per assignRoles); seat 1 is a composed bot for Domestic+Foreign.
  Choosing humanRole=foreign flips the assignment.
- 4-player solo + humanRole=domestic: seat 2 maps to human; the other
  three seats get the right bots.
- 1-player solo: zero bots; bot map is empty.
- Reloading mid-game preserves which seat is human.

## Dependencies

11.3, 11.4, 11.5, 11.6, 10.3 (lobby), 10.6 (creds).

## Out of scope

- Tunable bot difficulty.
- Mixing human + remote-human + bots in the same match. (Possible but
  defer.)
- Local-only solo without server. The GH Pages hot-seat path remains for
  ephemeral demos but does not record runs.
