# 01.4 — Headless test harness

**Parent:** [01-foundations.md](01-foundations.md)

## Goal

A single shared test factory used by every later stage's tests, so the
harness pattern doesn't drift. Models the existing approach in
[tests/game.test.ts](../tests/game.test.ts) but generalized.

## Files

- Create `tests/helpers/makeClient.ts`.
- Create `tests/helpers/runMoves.ts`.
- Create `tests/helpers/seed.ts`.
- Modify `tests/game.test.ts` to use the helpers.

## API

```ts
// tests/helpers/makeClient.ts
import { Client } from 'boardgame.io/client';
import type { Client as BClient } from 'boardgame.io/dist/types/src/client/client';
import type { SettlementState } from '../../src/game/types';

export interface MakeClientOptions {
  numPlayers?: 1 | 2 | 3 | 4;        // default 2
  seed?: string;                      // default 'test-seed'
  playerID?: string;                  // for spectator-style state inspection
}

export const makeClient = (opts?: MakeClientOptions): BClient<SettlementState>;

// tests/helpers/runMoves.ts
export interface MoveCall<S = SettlementState> {
  player: string;
  move: keyof Game<S>['moves'];
  args?: unknown[];
}
export const runMoves = (
  client: BClient<SettlementState>,
  calls: ReadonlyArray<MoveCall>,
): void;

// tests/helpers/seed.ts — produces a deterministic state for tests that
// don't want to drive setup; used by later stages that need a "halfway
// through a round" state.
export const seedAfterChiefDistribution = (
  partial?: Partial<SettlementState>,
): SettlementState;
```

## Steps

1. `makeClient` instantiates `Client({ game: Settlement, numPlayers, ... })`
   with the seed plumbed through bgio's random plugin (added in 02.3; for
   now, pass through whatever `Client` supports today).
2. `runMoves` iterates calls, switching the client's `playerID` per call so
   the `activePlayers` map (added in 02.2) is respected.
3. `seedAfterChiefDistribution` is intentionally tiny: a fixed
   `SettlementState` with the bank pre-emptied into circles. Real seeds
   grow as later stages need them.
4. Refactor the existing smoke test to call `makeClient` so we lock in the
   pattern.

## Tests

- `makeClient()` returns a fresh client with `getState().G` matching the
  `Settlement` setup output for `numPlayers=2`.
- `runMoves` with an invalid move call leaves state unchanged and surfaces
  the error.
- `seedAfterChiefDistribution()` produces equal states across two calls
  (immutability).
