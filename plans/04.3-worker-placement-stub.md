# 04.3 — Worker placement (stub)

**Parent:** [04-chief.md](04-chief.md)

## Goal

Declare the move surface for placing a worker on a Domestic-grid building
slot, gated behind a feature flag until 06 (Domestic) lands.

## Files

- Create `src/game/roles/chief/workerPlacement.ts`.
- Register move in `src/game/moves.ts`.
- Create `tests/roles/chief/workerPlacement.test.ts`.

## API

```ts
export const chiefPlaceWorker: Move<SettlementState>;
// args: { x: number; y: number }
```

## Steps

1. Validate caller holds chief and phase is `chiefPhase`.
2. **Feature gate:** if `G._features.workersEnabled !== true`,
   return `INVALID_MOVE`. The feature flag is set by 06's setup once
   `G.domestic.grid` exists.
3. Validate `(x,y)` corresponds to an existing in-play building.
4. Validate the chief has a free worker token in their reserve
   (`G.chief.workers > 0`).
5. Decrement `G.chief.workers`. Set `G.domestic.grid[(x,y)].worker =
   { ownerSeat: ctx.playerID }`.

## Tests

- Without 06 landed: move returns `INVALID_MOVE`.
- With a stubbed grid in tests + flag set: move places a worker, decrements
  reserve, marks grid cell.
- Placing on an empty cell returns `INVALID_MOVE`.
- Placing with no workers in reserve returns `INVALID_MOVE`.

## Dependencies

01.1, 02.1, 04.2 (chief still in their phase). Soft dep on 06 for the real
grid; until then, a stubbed grid is fine for tests.

## Out of scope

- Worker effects on production output (06.4).
- Removing / migrating workers between buildings.
