# Playtest notes — 2026-04-29

Session driven through Playwright MCP against `npm run dev` at
`http://localhost:5179/` (default hot-seat build, the same flavor that
ships to GitHub Pages today). Goal: try to actually play a game and
write down what blocks the user. No fixes here — this is a planning
input.

Screenshots referenced live in `notes/screenshots/`.

## TL;DR

In the hot-seat build (the default, the one served to GitHub Pages),
the app renders **no action surface unless the player digs into bgio's
built-in dev-only debug panel**. The Board treats the local viewer as
a spectator, all four role panels short-circuit, and the only thing
that responds to clicks is a "End my turn" stub at the bottom and the
chat composer (which has no transport behind it). That matches the
user's report: "I cannot get the game to do anything real."

The networked build is gated behind a lobby + accounts flow that
requires the server to be up; we haven't validated that path here.

## Blocking issues (would prevent any real session)

### B1. Hot-seat = spectator by default
**Where:** `src/App.tsx` HotSeatApp wires `Client({ ... })` with no
`playerID`. The board sees `props.playerID === undefined`. `Board.tsx`
treats that as `isSpectator` and passes it to the StatusBar; each
role panel (`ChiefPanel`, `SciencePanel`, `DomesticPanel`,
`ForeignPanel`) starts with `if (!playerID) return null`.

**Effect:** Default hot-seat URL renders no actions. The "End my
turn" stub at the bottom is the only clickable game control. To do
anything else you must open bgio's debug panel.

**Real fix shape:** hot-seat is one human controlling all seats; it
needs a seat picker baked into Board (e.g. a sticky tabs row "Play
as: Chief / Science / Domestic / Foreign") that updates a local
`viewerSeat` state and feeds it to the panels. bgio's `Client` also
accepts `playerID` and `client.updatePlayerID(...)` at runtime; we
should call that on the picker change so the engine's
`activePlayers` gating respects it.

### B2. No way to act in `othersPhase` from a single tab
**Where:** `othersPhase`'s `activePlayers` map pins each non-chief
seat to its own stage (`scienceTurn`/`domesticTurn`/`foreignTurn`).
A move from a non-active seat is rejected by bgio.

**Effect:** Even if B1 were fixed by exposing a seat picker, the user
would need to flip seats while bgio's engine state stays consistent.
The bgio debug panel's PLAYERS picker is what does that today, and
ONLY that — there's no app-level path.

### B3. `Mode: Spectating` is wrong in hot-seat
**Where:** `src/Board.tsx` reads `props.playerID == null` to decide
spectator. In hot-seat there's no playerID at all (no networked
seat), but the user is also not really watching someone else play.

**Effect:** Confusing label ("am I watching?"), reinforces the user's
sense that nothing is interactive. The label should be "Hot-seat" or
"Local" when the build mode is hot-seat.

### B4. Production build has no debug panel and no lobby
**Where:** `12.2` gated bgio's debug panel on `import.meta.env.DEV`.
`vite build` strips it.

**Effect:** the production GitHub Pages bundle has *literally zero*
action surface in hot-seat (no debug panel, no role panels, no seat
picker). The "Demo" promise on the README is currently broken — the
demo can't demo anything.

**Real fix shape:** in `vite build`, ship the seat picker (B1) so
hot-seat is playable end-to-end without the debug panel.

## Visual / UX issues

### V1. Two "End my turn" buttons during chiefPhase
Chief panel has its own "END MY TURN" footer button. Board.tsx ALSO
renders a legacy "End my turn" button at the page bottom. Both fire
a turn-ending move but the legacy one is always enabled, even
outside `chiefPhase`. The legacy button is a leftover from the 01.3
skeleton and should be removed once each role panel owns its own
"done" affordance.

### V2. Static role-assignment list duplicates engine state
Above the ChiefPanel, Board.tsx renders four "Player N: <role>" rows
that are pure info. They never react to the active turn (no
highlight, no disable). They feel like buttons but aren't.

### V3. CenterMat duplicates ChiefPanel circle info
When chief panel is visible, both the per-seat CircleEditors INSIDE
ChiefPanel and the standalone CenterMat row BELOW it show the same
"in circle: N" data. The CenterMat is the public read view for
non-chief seats; the chief panel is the chief's editor. Hide one or
the other based on local viewer.

### V4. CircleEditor only exposes gold
ChiefPanel's CircleEditor has `+1 / +2 / +5` for **gold only**. The
`chiefDistribute` move accepts any `Partial<ResourceBag>`. After 03
shipped 10 resources and 06 wired Domestic to consume non-gold
yields, the chief can't distribute anything that isn't gold via the
UI.

### V5. No "what's my next legal action" hint
There's no contextual instruction on screen. New player has no idea
what `chiefPhase` means, that they should distribute gold, that
"+1" sends 1 gold to that seat's circle, that End My Turn locks them
out.

### V6. No game-over or summary view
`endIf` returns `{ kind: 'win' | 'timeUp', ... }` and bgio surfaces
it on `ctx.gameover`. Nothing in Board.tsx reads that, so when the
game actually ends nothing visibly changes. (Not exercised in this
session but a guaranteed dead-end.)

### V7. Dev-only debug panel "1" button collides with bgio reset
bgio ships keyboard shortcut "1 → reset" in the CONTROLS row of the
debug panel. The PLAYERS row also has a button labeled "1". A casual
"click the player-1 button" (reasonable user behavior in a debug
panel) wipes the game state if you misclick. We can't fix bgio but
we can warn or move our seat-switching out of bgio's panel (B1).

### V8. Chat composer is wired in hot-seat but no transport
"No messages yet." renders forever; sent messages go nowhere. Wiring
chat in hot-seat is undefined per plan 10.5 (chat is a network
feature). Hide the chat in hot-seat.

### V9. Missing favicon — every load logs a 404
`http://localhost:5179/favicon.ico` returns 404. Cosmetic but
clutters the console.

### V10. Status bar truncates the role-assignment box
Top "Player 1: chief" rows visibly overlap the right-side bgio debug
panel (both float at the top). Looks fine on screenshot 01 but felt
cramped at 1400px viewport — narrower viewports could clip.

## Engine / model issues spotted while exploring

### E1. Cannot drive a real round to completion via the UI
Even granting B1/B2 are fixed, `othersPhase` requires the science /
domestic / foreign seats to play their respective moves and then
flip a "I'm done" flag (`G.othersDone[seat]`). No role panel
currently exposes a "done" affordance. The only way to advance
`othersPhase` today is the test-only `__testSetOthersDone` move,
which we just gated behind `NODE_ENV=test` (review fix #1). So the
engine's round model has no UI to drive it past chiefPhase.

### E2. Foreign panel's AssignDamageDialog stub returns `[{ byUnit: {} }]`
Stub auto-allocates 1 to a single unit. After review fix #7 the
shape is correct but the allocation is still wrong for any battle
where multiple units take damage in a round. Won't progress past
the first interesting battle.

### E3. ProductionMoves now omit `__test*` (review fix #1)
With those moves stripped from the production move set, the
*production* build has no way to advance through `othersPhase` at
all because the role-done plumbing isn't on any UI button. The
prior shipping behavior was buggier (clients could call __test
moves over the wire) but at least the move existed; we now have a
correctness improvement that surfaces a separate UI gap.

## Outside-the-app issues

### O1. README "Demo at the GitHub Pages URL" is currently misleading
A visitor following that link gets the page in screenshot 01 — no
controls. They will not figure out the bgio debug panel exists.

### O2. Lobby + auth is the only path to a "playable" build today
Networked mode requires `npm run server:dev` running on :8000, an
account registered, and a match created/joined. None of that has
been validated end-to-end via Playwright in this session. Worth a
follow-up playtest after the server's `/auth` routes (review fix
landed) plus SQLite (after `npm install` is run on the deploy host)
are exercised with two browser tabs.

## Plan-input punch list (priority order)

1. **Hot-seat seat picker.** Add a sticky "Play as: …" tab strip in
   `Board.tsx` that updates `client.updatePlayerID` and a local
   `viewerSeat` state. Each role panel reads viewerSeat instead of
   `playerID`. (Solves B1, B2, B4.)
2. **Per-role "End my role's turn" button.** SciencePanel /
   DomesticPanel / ForeignPanel each grow a footer button that sets
   `G.othersDone[seat] = true` via a real move. Remove the legacy
   bottom "End my turn" from Board.tsx. (Solves V1, E1, E3.)
3. **Mode label.** Status bar reads "Hot-seat" / "Networked" /
   "Spectating" based on the actual mode. (V3.)
4. **Hide CenterMat when ChiefPanel is visible.** Or merge the
   two views so the chief edits and everyone else watches the same
   widget. (V3.)
5. **Multi-resource CircleEditor.** Render a row per resource the
   chief actually has in the bank, not just gold. (V4.)
6. **Game-over banner.** Read `props.ctx.gameover` and render a
   "You won in N turns" / "Time's up: M settlements joined" overlay.
   (V6.)
7. **Hot-seat: hide chat.** Chat needs a transport; hot-seat doesn't
   have one. Render the chat pane only when `clientMode === 'networked'`.
   (V8.)
8. **Phase / move hints.** A small "What you can do now: …" line
   under the StatusBar driven by `(phase, stage, role)` triple. (V5.)
9. **Add favicon.** A trivial PNG to silence the 404. (V9.)
10. **README fix.** Either gate the demo claim on the seat picker
    landing, or replace it with "play locally with `npm run dev:full`".
    (O1.)
11. **AssignDamageDialog: real per-round absorber UI.** Resolver
    needs one allocation per incoming-damage event from enemy →
    player; the stub assumes one. (E2.)
12. **End-to-end networked playtest.** Two browser tabs, register
    alice + bob, create a match, both join, advance through one
    round. Capture similar notes. (O2.)

Items 1 and 2 are the unblockers — without them the game is
literally unplayable from the UI in either build flavor. Everything
else is incremental.
